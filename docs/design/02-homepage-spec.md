# Homepage Spec

This file defines the active homepage spec for Carbon Cut.

Use together with:

- `00-product-truth.md` for claim safety and capability boundaries
- `01-design-principles.md` for style/behavior direction

## Core goal

The homepage should help a visitor quickly understand:

- what Carbon Cut is now
- who it is for now
- what the current flow does
- what the user gets after completing the flow
- what to do next

## Positioning guardrails

Present Carbon Cut as:

- a testing-stage climate product
- a guided personal carbon-footprint assessment
- focused on practical understanding and first reduction steps

Do not present Carbon Cut as:

- an enterprise or team carbon platform
- audit/compliance reporting software
- a multi-audience mature workflow product

## Section structure

Default section order:

1. Hero
2. What Carbon Cut does
3. How the guided flow works
4. Result and recommendation preview
5. Product status/trust framing
6. Final CTA

Optional sections are allowed only if they improve clarity/trust without adding template-like filler.

## Content rules

- Claims must stay inside `00-product-truth.md` boundaries.
- Keep copy concrete and product-real, not hype-led.
- Do not invent unsupported capabilities to increase perceived maturity.

## UX rules for implementation

- Keep one primary CTA in key entry sections.
- Use familiar section rhythm and scannable hierarchy.
- Keep visual grouping clear and intentional.
- Preserve mobile and desktop coherence for the same core concept.
- Do not use feature-card overload as a substitute for narrative clarity.
- Do not use more than two card-heavy sections across the homepage without explicit approval.
- Do not use generic KPI-card grids as default homepage evidence/trust treatment.

## Build and review sequence

- Implement section-by-section (not full-page redesign in one pass by default).
- Validate each section against product truth and design principles.
- Run final pass for claim safety, accessibility, and responsive behavior.

## Acceptance checklist

- Matches current product truth and safe claim set.
- Feels specific to Carbon Cut, not generic climate-tech SaaS.
- Uses shared primitives/tokens consistently.
- Maintains clear hierarchy and CTA clarity.
- Works on desktop and mobile.
